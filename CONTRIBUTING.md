# 為 SDD Workflow 做出貢獻

感謝您有興趣為 SDD Workflow 專案做出貢獻！本文件提供了貢獻指南。

## 如何貢獻

### 回報問題

如果您發現 bug 或有建議：
1. 檢查問題追蹤器中是否已存在該問題
2. 建立新問題時請包含：
   - 清晰的標題和描述
   - 重現步驟（針對 bug）
   - 預期行為與實際行為
   - 範例 Gherkin/程式碼（如適用）

### 貢獻程式碼

1. Fork 此儲存庫
2. 建立功能分支（`git checkout -b feature/amazing-feature`）
3. 進行變更
4. 徹底測試您的變更
5. 使用描述性的 commit 訊息進行提交
6. 推送到您的 fork
7. 建立 Pull Request

### 貢獻範例

我們歡迎新範例！要新增範例：

1. 在 `examples/<your-feature>/` 下建立新目錄
2. 包含所有 4 個階段的輸出：
   - `spec.feature` - Gherkin 規格
   - `structure.py`（或 `.ts`）- 資料模型和介面
   - `implementation.py`（或 `.ts`）- 具體實作
   - `README.md` - 範例說明
3. 確保範例完整且可運作
4. 使用 SDD 工作流程指令測試它

### 改善文件

歡迎改善文件：
- 修正錯字或不清楚的說明
- 新增更多範例或使用案例
- 改善提示範本
- 翻譯文件（未來）

### 改善 Slash 指令

如果您想改善 slash 指令：
1. 徹底測試您的變更
2. 確保向後相容性
3. 更新文件
4. 提供改善後行為的範例

## 開發指南

### 撰寫提示

撰寫或改善提示時：
- 對約束條件要清晰明確
- 使用範例來說明概念
- 參考 expected_workflow.md 以保持一致
- 使用各種輸入測試提示

### 程式碼範例

新增程式碼範例時：
- 使用型別提示（Python）或嚴格型別（TypeScript）
- 遵循 SDD 理念：規格 → 結構 → 實作
- 包含將程式碼對應到 Gherkin 的註解
- 新增驗證/測試

### Gherkin 規格

撰寫 Gherkin 範例時：
- 嚴格遵循 Given-When-Then 格式
- 使用具體值，而非變數
- 涵蓋正常路徑、邊界案例和錯誤案例
- 保持情境聚焦且原子化

## 程式碼風格

### Python
- 遵循 PEP 8
- 使用型別提示
- 使用 Pydantic 建立資料模型
- 使用 ABC 建立介面

### TypeScript
- 使用嚴格的 TypeScript
- 遵循標準慣例
- 使用介面定義契約

### Gherkin
- Feature：句首大寫
- Scenario：句首大寫
- Steps：小寫（專有名詞除外）
- 縮排：2 個空格

## 測試

提交前：
1. 測試 slash 指令是否正常運作
2. 驗證範例能無錯誤執行
3. 檢查所有 4 個階段能協同運作
4. 驗證 Gherkin 語法

## Pull Request 流程

1. 更新任何變更功能的文件
2. 如適用，新增或更新範例
3. 確保所有範例仍能運作
4. 更新 CHANGELOG.md（如果存在）
5. 請維護者審查

## 程式碼審查

審查者會檢查：
- 與 SDD 理念的一致性
- 程式碼品質和清晰度
- 文件完整性
- 範例正確性
- 向後相容性

## 社群

- 保持尊重和建設性
- 幫助他人學習 SDD 方法
- 分享您的成功故事
- 回報什麼有效、什麼無效

## 有問題？

如果您有問題：
- 查看 README.md
- 審查現有範例
- 查看 expected_workflow.md
- 開啟討論問題

## 授權

透過貢獻，您同意您的貢獻將在 MIT 授權下授權。
